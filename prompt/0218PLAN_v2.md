# 0218PLAN_v2.md（新版方案｜双基数 + 可提前干预信号 + 闭环）

## 未成年人“风险监测中枢”（双基数：SEED+PEER；动态=视频夜游；案件链路仅静态底座）落地实施方案

### 一句话目标
以 **省厅口径严重不良未成年人对象池（SEED）** 为主基数，以 **GraphAny 关联扩圈“近墨者/高风险同伴（PEER）”** 为第二基数；在 **风险等级定义一致** 的前提下，通过 **差异化干预强度（SEED 强、PEER 轻）**，并以 **视频云人脸轨迹夜游（22:00–06:00）** 作为第一优先动态信号，实现 **“风险可见 → 可解释 → 可行动 → 可闭环 → 可迭代”**，目标导向明确：**让未成年人相关警情下降**。

---

## 0. 项目一句话定位（升级版：双基数+闭环）
> **“以省厅口径对象池为主、以同伴扩圈为辅；用可提前介入信号做动态预警，用案件链路做静态底座；以属地责任+时限+回执实现闭环治理。”**

核心关键词（给评委的“抓手”）：
- **双基数**：SEED（省厅严重不良）+ PEER（近墨者扩圈）
- **动态预警只用可提前干预信号**：夜游（视频轨迹）优先
- **四级分层一致**：LOW/MEDIUM/HIGH/CRITICAL
- **干预强度差异化**：SEED 更激进自动派单、更短时限；PEER 更强调研判与提醒，派单门槛更高
- **闭环**：预警 → 派单 → 回执 → 再评估 → 质效统计
- **可解释**：每条预警输出 `reasons_top3`（可直接写进派单内容）

## 0.1 总体架构（数据层/分析层/应用层 + 两个闭环）
### 0.1.1 三层架构
**数据层（Data）**
- SEED：`stdata.b_zdry_ryxx`（省厅口径种子库）
- PEER：GraphAny 输出（落地表/视图）
- 动态信号：视频云人脸轨迹 `ywdata.t_spy_ryrlgj_xx`（夜游）
- 静态底座：标签类表（留守/困境/辍学等）+ 既往违法链路（ZFBA/处罚/刑拘等，只做静态赋分）

**分析层（Analytics）**
- `static_score`：标签 + 既往违法链路（存量底座）+（对 PEER 增加 peer_score 映射）
- `dynamic_increment`：仅“可提前介入”信号（v1 只做夜游）
- `risk_level`：统一分段阈值（可配置）
- `reasons_top3`：按贡献度排序输出（可解释）

**应用层（App）**
- 风险中枢看板：双基数总览、TOP 风险清单、属地分布
- 个人画像：基础信息、标签、夜游轨迹摘要、同伴来源可追溯
- 派单闭环：预警/任务/回执/再评估/质效统计
- 配置与迭代：规则版本化、跑批日志、策略调整留痕

### 0.1.2 两个闭环
**预警闭环（业务闭环）**
1) 计算快照（risk_snapshot）  
2) 生成预警（alert）  
3) 自动/人工派单（task）  
4) 处置回执（feedback）  
5) 风险再评估（下一批快照对比 delta）  

**迭代闭环（治理闭环）**
- 误报原因归因 → 指标/阈值/动作矩阵调整 → 新规则集上线（rule_set 版本化）  
- 质效指标：按期反馈率、超期率、有效处置率、重复预警下降率、警情变化趋势（按属地/学校/区域）

### 2.2 GraphAny 输出落地：PEER 关系边（可追溯）
#### 2.2.1 GraphAny 输出表（必须包含可追溯字段）
**表名建议**：`ywdata.graphany_peer_edge`

字段要求（你提的最低标准，全部满足）：
- `person_key`（近墨者证件号）
- `source_seed_person_key`（触发关联的 SEED 证件号）
- `peer_score`（亲密度/强度，建议 0–1 或 0–100，需在配置中声明）
- `peer_reason`（解释文本或规则 id）
- `calc_time`（计算时间）

**DDL 示例：**
```sql
CREATE TABLE IF NOT EXISTS "ywdata"."graphany_peer_edge" (
  id                      uuid PRIMARY KEY,
  person_key              varchar(50) NOT NULL,
  source_seed_person_key  varchar(50) NOT NULL,
  peer_score              numeric(10,4) NOT NULL,
  peer_reason             text NULL,
  calc_time               timestamp without time zone NOT NULL,
  evidence_json           jsonb NULL,     -- 可选：证据（同驻次数/同框次数/地点等）
  create_time             timestamp without time zone NOT NULL DEFAULT CURRENT_TIMESTAMP
);

CREATE INDEX IF NOT EXISTS idx_peer_edge_person
ON "ywdata"."graphany_peer_edge"(person_key);

CREATE INDEX IF NOT EXISTS idx_peer_edge_seed
ON "ywdata"."graphany_peer_edge"(source_seed_person_key);

CREATE INDEX IF NOT EXISTS idx_peer_edge_calc
ON "ywdata"."graphany_peer_edge"(calc_time DESC);
```

#### 2.2.2 PEER 对象池视图：`ywdata.v_peer_minor`
> 逻辑：把“边”汇总为“人”。默认取 **最新一批 calc_time**，并聚合 peer_score / seed_count / 可解释来源。

```sql
CREATE OR REPLACE VIEW "ywdata"."v_peer_minor" AS
WITH latest AS (
  SELECT MAX(calc_time) AS calc_time
  FROM "ywdata"."graphany_peer_edge"
),
edges AS (
  SELECT
    UPPER(BTRIM(e.person_key))             AS person_key,
    UPPER(BTRIM(e.source_seed_person_key)) AS source_seed_person_key,
    e.peer_score,
    e.peer_reason,
    e.calc_time
  FROM "ywdata"."graphany_peer_edge" e
  JOIN latest l ON e.calc_time = l.calc_time
  WHERE NULLIF(BTRIM(e.person_key), '') IS NOT NULL
    AND NULLIF(BTRIM(e.source_seed_person_key), '') IS NOT NULL
)
SELECT
  person_key,
  MAX(peer_score)                                         AS peer_score,      -- 默认取 max；也可改 sum/avg（写入配置）
  COUNT(DISTINCT source_seed_person_key)                   AS seed_cnt,
  STRING_AGG(DISTINCT source_seed_person_key, '、')        AS seed_keys,
  STRING_AGG(DISTINCT COALESCE(peer_reason,''), '；')      AS peer_reason_text,
  MAX(calc_time)                                           AS calc_time
FROM edges
GROUP BY person_key;
```

### 2.3 统一人员维表：person_dim（双基数合并 + cohort_type）
**对象要求（文档必须包含）**：
- `person_key`
- `cohort_type`（SEED/PEER/BOTH）
- 基础信息（姓名/性别/出生等尽可能补齐）
- 属地（派出所/分局/市局编码与名称尽可能补齐）
- 标签集合（用于 static_score 与画像可解释）

**落地建议：视图 `ywdata.person_dim`**
- SEED 优先使用 `v_seed_minor` 的字段
- PEER-only 尝试从户籍/人口表补齐：`ywdata.t_dsfb_czrk_jbxx`（按 `gmsfhm=person_key` 取最新 `zxsj`）
- 组织名称映射：`ywdata.b_zzjgdm`

```sql
CREATE OR REPLACE VIEW "ywdata"."person_dim" AS
WITH seed AS (
  SELECT
    person_key,
    xm,
    xb,
    csrq,
    sssj_dm,
    ssfj_dm,
    sspcs_dm,
    hjdz,
    jzdz,
    lxdh,
    jjlxr,
    jjlxrdh,
    'SEED'::text AS cohort_flag
  FROM "ywdata"."v_seed_minor"
),
peer AS (
  SELECT
    person_key,
    peer_score,
    seed_cnt,
    seed_keys,
    peer_reason_text,
    calc_time,
    'PEER'::text AS cohort_flag
  FROM "ywdata"."v_peer_minor"
),
cohort AS (
  SELECT person_key FROM seed
  UNION
  SELECT person_key FROM peer
),
hj AS (
  SELECT DISTINCT ON (aa.gmsfhm)
    UPPER(BTRIM(aa.gmsfhm)) AS person_key,
    aa.xm                   AS hj_xm,
    aa.xbdm                 AS hj_xb,
    aa.csrq                 AS hj_csrq,
    aa.hjdz_qhnxxdz         AS hj_hjdz,
    aa.sjgsdwdm             AS hj_sspcsdm,
    aa.zxsj                 AS hj_zxsj
  FROM "ywdata"."t_dsfb_czrk_jbxx" aa
  WHERE NULLIF(BTRIM(aa.gmsfhm), '') IS NOT NULL
  ORDER BY aa.gmsfhm, aa.zxsj DESC NULLS LAST
),
org AS (
  SELECT sspcsdm, sspcs, ssfjdm, ssfj, sssjdm, sssj
  FROM "ywdata"."b_zzjgdm"
),
tag AS (
  -- 统一标签：存在即 1（可扩展）
  SELECT person_key, 'LS' AS tag_code FROM "ywdata"."b_per_qslswcnr"
  UNION ALL SELECT person_key, 'KJ' FROM "ywdata"."b_per_qskjwcnr"
  UNION ALL SELECT person_key, 'CX' FROM "ywdata"."b_per_qscxwcnr"
  UNION ALL SELECT person_key, 'YBBL' FROM "ywdata"."b_per_yfsjyjblxwwcnr"
  UNION ALL SELECT person_key, 'YZJSZA' FROM "ywdata"."b_per_qsyzjszawcnr"
  UNION ALL SELECT person_key, 'ZLJSCJ' FROM "ywdata"."b_per_qszljscjwcnr"
  UNION ALL SELECT UPPER(BTRIM(sfzjh)) AS person_key, 'LD' FROM "ywdata"."v_per_qsldwcnrxs"
),
tag_agg AS (
  SELECT
    UPPER(BTRIM(person_key)) AS person_key,
    STRING_AGG(DISTINCT tag_code, ',') AS tag_codes
  FROM tag
  WHERE NULLIF(BTRIM(person_key), '') IS NOT NULL
  GROUP BY UPPER(BTRIM(person_key))
)
SELECT
  c.person_key,
  CASE
    WHEN s.person_key IS NOT NULL AND p.person_key IS NOT NULL THEN 'BOTH'
    WHEN s.person_key IS NOT NULL THEN 'SEED'
    ELSE 'PEER'
  END AS cohort_type,

  -- 基础信息：SEED 优先，否则用户籍补齐，否则用空
  COALESCE(s.xm, hj.hj_xm) AS xm,
  COALESCE(s.xb, hj.hj_xb) AS xb,
  COALESCE(s.csrq, hj.hj_csrq) AS csrq,

  -- 属地：SEED 优先，否则用人口表 sjgsdwdm（派出所代码）
  COALESCE(s.sspcs_dm, hj.hj_sspcsdm) AS sspcsdm,
  o.sspcs AS sspcs_name,
  o.ssfjdm,
  o.ssfj AS ssfj_name,
  o.sssjdm,
  o.sssj AS sssj_name,

  COALESCE(s.hjdz, hj.hj_hjdz) AS hjdz,
  s.jzdz,
  s.lxdh, s.jjlxr, s.jjlxrdh,

  -- PEER 扩圈信息（仅 PEER/BOTH 有）
  p.peer_score,
  p.seed_cnt,
  p.seed_keys,
  p.peer_reason_text,
  p.calc_time AS peer_calc_time,

  t.tag_codes
FROM cohort c
LEFT JOIN seed s ON c.person_key = s.person_key
LEFT JOIN peer p ON c.person_key = p.person_key
LEFT JOIN hj ON c.person_key = hj.person_key
LEFT JOIN org o ON COALESCE(s.sspcs_dm, hj.hj_sspcsdm) = o.sspcsdm
LEFT JOIN tag_agg t ON c.person_key = t.person_key;
```

### 2.4 事件总线：event_bus（动态信号承载；v1 至少支持夜游）
> 事件总线原则：**动态只收“可提前介入”信号**。案件链路不进 event_bus（它是静态底座）。

#### 2.4.1 event_bus 表（可扩展）
```sql
CREATE TABLE IF NOT EXISTS "ywdata"."event_bus" (
  event_id          uuid PRIMARY KEY,
  person_key        varchar(50) NOT NULL,
  event_type        varchar(30) NOT NULL,           -- VIDEO_NIGHT / PHONE_COLOC / SCHOOL_ABN / CROSS_DISTRICT ...
  event_time        timestamp without time zone NOT NULL,
  location_id       varchar(80) NULL,               -- device_id / 场所编码 / 学校编码等
  district_code     varchar(30) NULL,               -- 派出所/区县等（可选）
  source_table      varchar(80) NULL,
  raw_pk            varchar(120) NULL,
  ext_json          jsonb NULL,
  ingest_time       timestamp without time zone NOT NULL DEFAULT CURRENT_TIMESTAMP
);

CREATE INDEX IF NOT EXISTS idx_event_bus_person_time
ON "ywdata"."event_bus"(person_key, event_time DESC);

CREATE INDEX IF NOT EXISTS idx_event_bus_type_time
ON "ywdata"."event_bus"(event_type, event_time DESC);
```

#### 2.4.2 视频夜游事件视图（从现有轨迹表解析）
> 本库已存在：`ywdata.t_spy_ryrlgj_xx(id_number, shot_time, device_id, directorycode, ...)`  
> `shot_time` 为 varchar，需要按实际格式解析；v1 默认按 “YYYY-MM-DD HH24:MI:SS” 解析，不符则在“字段映射清单”中确认并替换解析表达式。

```sql
CREATE OR REPLACE VIEW "ywdata"."v_video_face_hit" AS
SELECT
  UPPER(BTRIM(t.id_number)) AS person_key,
  -- {shot_time_parse}：按 shot_time 实际格式替换
  to_timestamp(t.shot_time, 'YYYY-MM-DD HH24:MI:SS') AS event_time,
  t.device_id AS location_id,
  t.directorycode AS district_code,
  'ywdata.t_spy_ryrlgj_xx' AS source_table,
  t.face_id AS raw_pk
FROM "ywdata"."t_spy_ryrlgj_xx" t
WHERE NULLIF(BTRIM(t.id_number), '') IS NOT NULL
  AND NULLIF(BTRIM(t.shot_time), '') IS NOT NULL;
```

```sql
CREATE OR REPLACE VIEW "ywdata"."v_video_night_event" AS
SELECT
  person_key,
  event_time,
  location_id,
  district_code,
  source_table,
  raw_pk
FROM "ywdata"."v_video_face_hit"
WHERE (EXTRACT(HOUR FROM event_time) >= 22 OR EXTRACT(HOUR FROM event_time) < 6);
```

> 说明：  
> - v1 可直接用 `v_video_night_event` 做聚合指标，不强制入库到 `event_bus`。  
> - 若轨迹量大、解析开销高，则在 v1 后半程将 `v_video_night_event` 增量落地到 `event_bus`（跑批/定时任务实现）。

### 2.6 预警闭环三表：alert / task / feedback

## 3. 风险引擎（static_score + dynamic_increment；输出可解释 reasons_top3）
### 3.0 关键改造点2：为什么“案件链路只做静态底座”
公安预防的目标是 **“警情下降、风险前移”**。  
ZFBA/案件/处罚/刑拘属于 **违法犯罪事实发生后的后链路**，它对“谁是高风险人”有价值，但不适合作为“动态触发信号”，原因：
- **时效性滞后**：案件已发生，动态预警再触发就容易变成“事后统计”
- **干预窗口已过**：真正能降低警情的是“事前介入”而不是“事后追责”
- **派出所感知差**：如果动态指标都是“已发生事实”，基层会认为系统只是在复述结果

因此：  
- **案件链路（ZFBA/处罚/刑拘/文书）只进入 `static_score`**，作为“存量风险底座”；  
- **动态预警（`dynamic_increment`）只使用可提前介入信号**，v1 只做“视频夜游”。

---

### 3.1 static_score（静态底座规则：标签 + 既往违法链路 + PEER关系强度）
#### 3.1.1 标签赋分（示例，可配置）
- 辍学（CX）：+25
- 困境（KJ）：+15
- 留守（LS）：+10
- 一般不良（YBBL）：+10
- 流动（LD）：+10
- 严重精障（YZJSZA）：+10
- 智精残疾（ZLJSCJ）：+10

#### 3.1.2 既往违法链路（示例，可配置；仅静态）
> 统计窗口建议：近 180 天 / 365 天（可配置），用于体现“存量风险”。

- 近365天涉案次数：每起 +5（封顶 +20）
- 近365天涉刑涉案次数：每起 +10（封顶 +30）
- 近365天处罚/刑拘命中：按严重度加分（封顶 +40）

> 注意：即便静态底座高，也不等于必须强派单——强弱由“二维策略矩阵”控制。

#### 3.1.3 PEER 关系强度映射（PEER 专属静态项）
> 目的：让“近墨者”即使没有既往案事件，也能体现“被带偏风险”。

- `peer_score_points = map(peer_score)`  
  - 若 peer_score 为 0–1：`peer_score_points = peer_score * 20`（封顶 20）  
  - 若 peer_score 为 0–100：`peer_score_points = (peer_score/100) * 20`

- `seed_cnt_points = min(10, seed_cnt * 2)`（连接多个 SEED 的人更危险）

---

### 3.2 dynamic_increment（动态增量：只用视频夜游）
#### 3.2.1 7天/30天指标（必须输出到 metrics_json）
- `night_hit_7d`：近7天夜间命中次数（22:00–06:00）
- `night_hit_30d`：近30天夜间命中次数
- `night_device_7d`：近7天夜间涉及设备数（可选）
- `night_active_days_7d`：近7天夜间活跃天数（可选）
- `night_hit_trend`：近7天相对前7天增幅（可选，体现“风险增量”）

#### 3.2.2 积分规则（示例，可配置 + 封顶）
- `night_hit_7d * 4`（封顶 +40）
- `night_active_days_7d * 3`（封顶 +15）
- `night_device_7d * 2`（封顶 +10）
- 动态总封顶：`dynamic_increment <= 60`

> v1 建议：先用 `night_hit_7d` + `night_hit_30d` 两个指标跑通闭环；其余指标作为增强项。

---

### 3.3 风险等级分段阈值（两层基数一致；可配置）
- 0–29：LOW（关注记录）
- 30–59：MEDIUM（电话提醒家长/提醒研判）
- 60–79：HIGH（属地见面核查/强提醒）
- ≥80：CRITICAL（家校联动 + 家庭干预/联合处置）

---

### 3.4 输出必须可解释：reasons_top3（用于派单内容）
`reasons_top3`（jsonb）每条至少包含：
- `code`：指标编码（如 `TAG_CX`、`DYN_NIGHT_7D`、`PEER_SCORE`）
- `label`：可读解释（如“辍学标签命中”“近7天夜间命中次数”）
- `value`：原始值（次数/分值）
- `points`：该项贡献分

示例：
```json
[
  {"code":"DYN_NIGHT_7D","label":"近7天夜间命中次数(22:00-06:00)","value":8,"points":32},
  {"code":"TAG_CX","label":"失学辍学标签命中","value":1,"points":25},
  {"code":"PEER_SCORE","label":"近墨者关系强度","value":0.86,"points":17.2}
]
```

---

### 3.5 干预策略（改造点1：二维策略映射表｜cohort_type × risk_level）
> 关键要求：**风险等级定义一致**，但 **干预强度不同**。  
> SEED：强干预（更激进自动派单、更短时限）  
> PEER：轻干预（更多人工研判/提醒，派单门槛更高）

#### 3.5.1 二维策略映射表（建议默认）
| cohort_type | risk_level | action（动作） | SLA | 自动派单 | 派单渠道优先级 | 短信兜底 |
|---|---|---|---:|---|---|---|
| SEED | LOW | 关注记录（建档关注） | - | 否 | - | 否 |
| SEED | MEDIUM | 电话提醒家长（或监护人） | 48h | 是 | 上级任务接口 →（失败）短信 → 手工 | 是（限频） |
| SEED | HIGH | 属地见面核查 | 24–48h | 是 | 上级任务接口 →（失败）短信 → 手工 | 是（限频） |
| SEED | CRITICAL | 家校联动+家庭干预（联合处置） | 24h | 是 | 上级任务接口 →（失败）短信 → 手工 | 是（限频） |
| PEER | LOW | 关注记录（纳入研判池） | - | 否 | - | 否 |
| PEER | MEDIUM | 提醒研判（不触达家长为主） | 72h | 否 | 手工（研判后再派） | 否 |
| PEER | HIGH | 电话提醒/研判（视情况） | 72h | 否（默认） | 手工（研判后派） | 否 |
| PEER | CRITICAL | 属地见面核查（重点核实同伴链条） | 48h | 是（门槛更高） | 上级任务接口 →（失败）手工 | 否（默认） |

#### 3.5.2 “SEED 强于 PEER”的具体体现（写给评委）
- **同一 risk_level** 下：SEED 更短 SLA、更高自动派单比例、更强处置动作
- **PEER 以研判为前置**：多数情况下先研判再派单，减少误伤与扰民
- **触达策略不同**：PEER 默认不短信、不直接家长触达；SEED 才启用短信兜底（且限频）

### A-2 视频夜游聚合 SQL（22:00–06:00；7天/30天）
> 基于现有表：`ywdata.t_spy_ryrlgj_xx(id_number, shot_time, device_id, directorycode, ...)`  
> `{shot_time_parse}` 按实际格式替换。

```sql
WITH ev AS (
  SELECT
    UPPER(BTRIM(t.id_number)) AS person_key,
    to_timestamp(t.shot_time, 'YYYY-MM-DD HH24:MI:SS') AS ts,
    t.device_id
  FROM "ywdata"."t_spy_ryrlgj_xx" t
  WHERE NULLIF(BTRIM(t.id_number),'') IS NOT NULL
    AND NULLIF(BTRIM(t.shot_time),'') IS NOT NULL
),
night AS (
  SELECT *
  FROM ev
  WHERE (EXTRACT(HOUR FROM ts) >= 22 OR EXTRACT(HOUR FROM ts) < 6)
)
SELECT
  p.person_key,
  COUNT(*) FILTER (WHERE ts >= (CURRENT_TIMESTAMP - INTERVAL '7 day'))  AS night_hit_7d,
  COUNT(*) FILTER (WHERE ts >= (CURRENT_TIMESTAMP - INTERVAL '30 day')) AS night_hit_30d,
  COUNT(DISTINCT device_id) FILTER (WHERE ts >= (CURRENT_TIMESTAMP - INTERVAL '7 day')) AS night_device_7d,
  COUNT(DISTINCT DATE(ts)) FILTER (WHERE ts >= (CURRENT_TIMESTAMP - INTERVAL '7 day'))  AS night_active_days_7d
FROM "ywdata"."person_dim" p
LEFT JOIN night n ON p.person_key = n.person_key
GROUP BY p.person_key;
```

### A-3 风险快照 + 预警生成（伪代码/SQL 思路）
> 目的：体现“闭环可落地”，并把二维矩阵落到 task 生成逻辑中。

**伪代码（推荐写法，便于落地到 Flask/service/dao）**
```text
batch_id = uuid()
as_of = now()

people = SELECT * FROM person_dim

for person in people:
  static_score = calc_static(tags, zfba_history, peer_score)
  dynamic_inc  = calc_dynamic(night_hit_7d, night_hit_30d, ...)
  risk_score   = min(100, static_score + dynamic_inc)
  risk_level   = bucket(risk_score, thresholds)

  reasons_top3 = top3_contributors(...)
  INSERT risk_snapshot(batch_id, person_key, cohort_type, scores, level, reasons, metrics, as_of)

  action = matrix[cohort_type][risk_level]
  if action.auto_dispatch:
     if not has_open_alert(person_key):

        DISPATCH to upstream API
        if fail and action.sms_fallback:
           SEND SMS (rate-limit)
```
